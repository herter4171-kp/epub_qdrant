# Bedrock Compare

A retrieval benchmark that stress-tests three RAG backends against 50 structured prompts,
then uses an LLM to generate and judge responses from each source.

---

## The Meta Loop: Using the Knowledge Base to Build Itself

This benchmark is the closing step of a bootstrapping process where the corpus we were
building was also the primary tool we used to figure out how to build it better.

The sequence went roughly like this:

**1. Start with what we knew.**
We stood up a basic Qdrant store and loaded it with books on agentic AI — standard dense
embeddings, nothing fancy. The goal was just to have something queryable.

**2. Query the books to learn what to do next.**
With a working (if rudimentary) retrieval system, we could ask it questions about
embedding strategies, chunking approaches, and vector architecture. The books told us
what the established next steps were — better embeddings, hybrid vectors, semantic
chunking. We used the knowledge base to design its own upgrade.

**3. Upgrade the books.**
Armed with that guidance, we rebuilt the book collection with hybrid dense + sparse
vectors and proper semantic chunking. The books went from being a rough index to being
a high-fidelity retrieval target.

**4. Use the upgraded books to figure out how to handle PDFs.**
Academic papers are structurally messier than books — equations, tables, multi-column
layouts, citation noise. With the book retrieval working well, we could query it for
guidance on PDF parsing pipelines (MinerU, layout-aware chunking, etc.) and actually
get useful answers back.

**5. Upgrade the papers.**
We applied what we learned to ingest the academic papers corpus with the same hybrid
vector treatment. The papers collection went from a dense-only index to a properly
structured hybrid store.

**6. Now measure whether any of it worked.**
That's this benchmark. We're querying the upgraded papers collection (`papers-semantic`)
against the original dense-only papers collection (`papers`) and against Amazon Bedrock's
managed KB as an external reference point. The question is whether the iterative
improvements to ingestion and retrieval actually show up as better scores — and whether
the knowledge we extracted from the corpus to guide those improvements was itself
well-retrieved.

The test is both a retrieval benchmark and a sanity check on the whole loop. If
`papers-semantic` doesn't meaningfully outperform `papers` on the prompts that were
designed to stress hybrid retrieval (especially `lexical_vs_semantic`), the upgrade
didn't land. If it does, the bootstrapping worked.

---

## What This Test Does

We wanted to know whether our Qdrant-based retrieval (dense and hybrid) actually outperforms
Amazon Bedrock's managed Knowledge Base on realistic agentic queries. Rather than synthetic
benchmarks, we built prompts that simulate how a real user would explore a corpus — asking
for adjacent chunks, filtering by heading, comparing keyword vs. semantic results, and
tracing cross-document lineage.

The test runs in three stages:

1. **Retrieve** — run all 50 prompts against all three sources at multiple top-k values
2. **Respond** — feed each source's retrieved chunks to an LLM and generate a response
3. **Judge** — score each response on retrieval quality and response quality (1–10 each)

---

## How the Prompts Were Made

`prompt_inspo.md` was generated by Gemini Deep Research as a long-form essay on agentic
retrieval architectures. It included a 50-prompt evaluation matrix organized by category
and proficiency level (1–5). We used that as the raw surface for prompt coverage.

`prompts.json` is the refined version — prompts rewritten to sound like cold-start user
messages with no assumed context, organized into 5 categories × 10 prompts each (2 per
proficiency level).

### Prompt Categories

| Category | Prompts | What it tests |
|---|---|---|
| `spatial_orientation` | 1–10 | Navigating adjacent chunks by index |
| `structural_aggregation` | 11–20 | Filtering and traversing heading hierarchy |
| `lexical_vs_semantic` | 21–30 | Dense vs sparse signal differentiation |
| `corpus_lineage` | 31–40 | Cross-document metadata traversal |
| `analytical_deep_dives` | 41–50 | Token density, metadata auditing, context management |

Proficiency levels 1–5 map to: Aware → Developing → Proficient → Advanced → Expert.
Higher proficiency prompts require more sophisticated tool-calling to answer correctly.

---

## Retrieval Sources

| Source | Collection | Type | Notes |
|---|---|---|---|
| `papers` | Qdrant `papers` | Dense only | 768-d Ollama embeddings, 90K points |
| `papers_semantic` | Qdrant `papers-semantic` | Hybrid dense + sparse | MiniCOIL sparse, 117K points |
| `bedrock` | AWS KB `RQMBIXUSXH` | Managed vector search | `us-gov-west-1`, no reranking |

`papers_semantic` receives `topk * 2` results (half sparse, half dense) so the LLM can
reason about both signals. The other two sources receive `topk` results.

## Pipeline

### Step 1 — Retrieve (`query_all.py`)

```bash
# Run all 50 prompts at topk values 2, 4, 8, 16, 32
bash 0_retrieve_using_prompts.sh

# Single prompt (debugging)
python3 query_all.py --topk 8 --prompt-id 5

# Validate connections without writing files
python3 query_all.py --topk 4 --dry-run
```

Requires:
- `agent-lookup` MCP server running on `localhost:8090`
- `papers-bedrock` MCP server via `uvx awslabs.bedrock-kb-retrieval-mcp-server@latest`
- AWS credentials for `us-gov-west-1`

Output: `query_results/{id}_{category}_{proficiency}_{topk}.json`

Each file contains the raw retrieval payload from all three sources plus timing metadata.

### Step 2 & 3 — Respond + Judge (`invoke_llm_trials.py`)

```bash
python3 invoke_llm_trials.py \
  --prompt-dense system_prompts/prompt_dense.txt \
  --prompt-semantic system_prompts/prompt_semantic.txt \
  --prompt-judge system_prompts/prompt_judge.txt \
  --model gemma4 \
  --limit 10        # omit to run all
  --overwrite       # re-run files that already exist
```

This script runs both phases in a single pass per file:

- **Harvest phase**: calls the LLM once per source using the appropriate system prompt,
  writes to `query_responses/`
- **Judge phase**: calls the LLM judge with all three responses and their retrieval context,
  writes scored assessments to `response_assessment/`

The `papers` and `bedrock` sources use `prompt_dense.txt`. The `papers_semantic` source
uses `prompt_semantic.txt`, which includes additional instructions for interpreting the
split sparse/dense result set.

#### Deprecated scripts

`harvest_responses.py` and `judge_responses_for_runs.py` ran these two phases separately.
They are superseded by `invoke_llm_trials.py` and should not be used.

## System Prompts

| File | Used for | Key behavior |
|---|---|---|
| `system_prompts/prompt_dense.txt` | `papers`, `bedrock` | Treat results as factual, cite inline, flag when search fails |
| `system_prompts/prompt_semantic.txt` | `papers_semantic` | Same as dense, plus: first half of results are sparse (good for exact terms), second half are dense (good for conceptual queries) |
| `system_prompts/prompt_judge.txt` | Judge LLM | Cold-start rule (no assumed context), scores retrieval and response independently 1–10, outputs JSON only |

## Final Outputs

### `query_results/`

Raw retrieval payloads. One file per (prompt, topk) combination.

```
{id}_{category}_{proficiency}_{topk}.json
```

```json
{
  "id": 1,
  "category": "spatial_orientation",
  "proficiency": 1,
  "prompt": "...",
  "topk": 8,
  "timestamp": "...",
  "sources": {
    "papers": { "content": [...], "lookup_time": 0.4 },
    "papers_semantic": { "content": [...], "lookup_time": 0.6 },
    "bedrock": { "content": [...], "lookup_time": 1.2 }
  }
}
```

### `query_responses/`

LLM-generated responses per source. Same filename as the input.

```json
{
  "input_file": "1_spatial_orientation_1_8.json",
  "prompt": "...",
  "responses": {
    "papers": {
      "response_text": "...",
      "model": "gemma4",
      "elapsed_seconds": 4.2,
      "error": null
    },
    "papers_semantic": { ... },
    "bedrock": { ... }
  }
}
```

### `response_assessment/`

Judge scores for each source. This is the primary analysis artifact.

```json
{
  "input_file": "1_spatial_orientation_1_8.json",
  "category": "spatial_orientation",
  "proficiency": 1,
  "topk": 8,
  "prompt": "...",
  "responses": { ... },
  "scores": {
    "papers": {
      "retrieval_score": 1,
      "retrieval_basis": "...",
      "response_score": 1,
      "response_basis": "..."
    },
    "papers_semantic": { ... },
    "bedrock": { ... },
    "source_verdict": {
      "title": "...",
      "source_file": "..."
    }
  },
  "judge_model": "gemma4",
  "judge_elapsed_seconds": 11.6,
  "judged_at": "..."
}
```

Each source gets two scores:
- `retrieval_score` (1–10): how useful were the retrieved chunks for answering the prompt
- `response_score` (1–10): how well did the LLM use what it was given

Scores are **not** comparable across sources — each source has a different index architecture.
The judge is instructed to evaluate each source independently.

## Suggested Analysis Routes (via `agent-lookup` on `papers-semantic`)

The `response_assessment/` files give you per-prompt, per-source, per-topk scores. Here are
concrete directions for deeper analysis, each grounded in what the academic literature covers.

Run these queries against the `papers-semantic` collection using the MCP `query` tool or
the `mcp_agent_lookup_query` tool in Kiro:

### 1. Score degradation by proficiency level

**Question**: Do all three sources degrade at the same rate as prompt complexity increases
from level 1 to 5, or does one source hold up better on expert-level queries?

```
query: "retrieval performance degradation complex multi-hop queries agentic"
collection: papers-semantic
mode: answer
```

Aggregate `retrieval_score` and `response_score` by `proficiency` and `category` across
all topk values. Plot mean score vs. proficiency level per source.

### 2. Optimal topk per category

**Question**: Does `spatial_orientation` (which relies on metadata traversal, not semantic
similarity) benefit from higher topk, or does it plateau early?

```
query: "optimal top-k retrieval chunk count context window RAG"
collection: papers-semantic
mode: answer
```

For each category, plot mean `retrieval_score` vs. topk (2, 4, 8, 16, 32). Expect
`lexical_vs_semantic` to show the most sensitivity to topk.

### 3. Bedrock vs. Qdrant on exact-term queries

**Question**: On `lexical_vs_semantic` prompts (which explicitly ask for exact keyword
matching), does `papers_semantic` outperform `bedrock` on `retrieval_score`?

```
query: "sparse vector BM25 exact keyword retrieval versus dense semantic search comparison"
collection: papers-semantic
mode: answer
```

Filter `response_assessment/` to `category == "lexical_vs_semantic"`, compare
`retrieval_score` distributions across sources.

### 4. Cold-start failure modes

**Question**: The judge prompt enforces a cold-start rule — responses that fabricate a
document reference are capped at response_score ≤ 4. How often does each source trigger
this penalty, and on which categories?

```
query: "hallucination fabrication RAG grounding faithfulness evaluation"
collection: papers-semantic
mode: answer
```

Scan `raw_judge_response` fields for phrases like "fabricated" or "assumed context".
Cross-reference with `response_score` ≤ 4 to find the failure distribution.

### 5. Source verdict concentration

**Question**: The judge emits a `source_verdict` naming the single most useful source per
prompt. Which source wins most often, and does it vary by category or proficiency?

No literature query needed — this is a straight aggregation over `scores.source_verdict`
across all assessment files.

### 6. Retrieval vs. response score gap

**Question**: When `retrieval_score` is low but `response_score` is high, the LLM
gracefully handled a bad retrieval (asked for clarification, flagged the failure). When
both are low, the LLM hallucinated. Mapping this gap reveals which source has the best
failure-handling behavior.

```
query: "graceful degradation retrieval failure LLM response quality RAG"
collection: papers-semantic
mode: answer
```

Compute `response_score - retrieval_score` per file per source. Positive delta = good
failure handling. Negative delta = hallucination under bad retrieval.
